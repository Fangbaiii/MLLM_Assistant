# AGENT.md

## 目标
- 保持高内聚、低耦合、可测试、可替换。
- 页面层薄，业务逻辑下沉，类型边界清晰。
- 优先做最小闭环改动，不做与当前任务无关的重构。

## 目录约定
- `src/app/`：只放路由入口、布局和 API route。这里负责装配，不承载复杂业务。
- `src/server/`：放服务端能力，包括上传、解析、第三方适配、mock response builder 等。
- `src/components/<feature>/`：按功能拆分 UI 组件，同一组件只负责一个明确职责。
- `src/components/ui/`：放纯通用原子组件，不掺杂业务规则。
- `src/lib/`：放纯函数、协议适配、API 客户端、与 React 无关的共享逻辑。
- `src/store/`：按领域拆分状态。
- `src/store/chat-session-store.ts`：只管理会话、消息、模式、流式更新。
- `src/store/evidence-store.ts`：只管理上传文件、OCR、证据文档。
- `src/store/workbench-store.ts`：只管理工作台偏好、主题、面板开关、面板尺寸。
- `src/mocks/`：只放 demo 数据和 mock 行为，不混入生产逻辑。
- `src/types/`：按领域拆分类型，避免单个类型文件持续膨胀。

## 编码原则
- 一个文件只负责一类职责。
- 能复用的逻辑必须抽取，避免在多个组件中复制粘贴。
- Store 只暴露语义化 action，不要在组件里绕过封装直接改状态。
- API route 只做输入校验、调用 service、组织响应；复杂逻辑放进 `src/server/`。
- 新功能优先沿现有边界扩展，不要把 chat、upload、evidence 强耦合到同一层。
- 工作台布局统一复用共享的分栏与拖拽能力，避免每个页面各写一套 resize 逻辑。

## 前后端边界
- 前端组件只关心展示、交互、状态组合。
- 后端能力通过 `src/app/api/*` 暴露接口，通过 `src/server/*` 实现细节。
- 与上传、OCR、模型调用相关的逻辑不得直接塞回页面组件。
- 出问题时应能快速判断是页面层、状态层、接口层还是服务层。

## UI 文案要求
- 禁止在产品 UI 中保留草稿式、说明式、需求式文案。
- 禁止出现类似“像 VSCode 一样保留清晰工作流”这类开发过程文本。
- UI 文案必须面向最终用户，简洁、明确、可直接使用。

## 交互约定
- 左侧会话栏、右侧证据栏、底部回答框都应保持可独立拖拽调节尺寸。
- 拖拽能力优先沉淀为共享组件或共享模式，不要散落在业务组件里重复实现。
- 工作流要清晰，结构要像专业 IDE 一样稳定，但不要把设计要求直接写进界面文案。

## 修改前检查
- 先阅读相关文件，确认已有边界和复用点。
- 只修改和当前任务直接相关的文件。
- 如果要引入新概念，先判断它属于 `app / server / components / lib / store / mocks / types` 的哪一层。

## 交付标准
- `CI=true pnpm lint` 必须通过。
- 涉及运行时逻辑时，再补 `pnpm build`。
- 代码应当可读、可维护，避免为了能跑而堆叠判断和特例。

## 对后续 AI 的硬要求
- 不要重写无关文件。
- 不要把 mock 代码直接混进生产逻辑。
- 不要在组件里塞大段业务规则。
- 不要把服务端逻辑放回 `src/lib` 或页面组件。
- 遇到架构不确定时，优先抽函数、抽服务、抽类型，而不是继续膨胀当前文件。
